# routing-regression-results（真实回归测试结果）

## 结论先说
这一轮先做的是**规则级真实回归**，不是模型盲跑分数游戏。
也就是说，我先验证当前 skill 的**文种边界、输出模式桥接、部门口径路由、rewrite/review 链入口**，是否已经具备支撑真实调用的结构完整性。

当前判断：
- **主干已基本通过**；
- **高频场景可用性较强**；
- **边界题判断能力仍需继续显式化**；
- **rewrite / review 链已经具备接入条件，但还没把部门口径下沉到位。**

---

## 一、已通过的部分

### 1. 文种主边界已可支撑真实调用
当前规则层已经能清楚区分：
- 请示 vs 报告
- 通知 vs 通报
- 会议纪要 vs 会议记录
- 制度 / 办法 vs 方案
- 函 vs 通知

这意味着在多数真实场景下，skill 已经具备“先分对文种再起稿”的基础，不再只是堆模板。

### 2. 部门口径主边界已经成型
当前规则层已能稳定区分：
- 行政：统一安排 / 周知执行 / 报送衔接
- 人事：人员管理 / 培训纪律 / 岗位责任
- 后勤：现场保障 / 设施设备 / 安全整改
- 综合运营：任务推进 / 节点管理 / 协同闭环

这意味着高频内部文种已经不必继续靠“凭感觉切部门”。

### 3. 高批次 drafting prompt 已基本吃进部门口径
当前已完成部门口径下沉的 prompt 包括：
- 通知
- 情况说明 / 总结
- 工作计划
- 报告
- 请示
- 通报
- 会议纪要
- 制度
- 决定

这说明 skill 已经从“参考层强”推进到“执行层开始真正受益”。

---

## 二、当前仍有缺口的部分

### 1. 边界题还缺更显式的回归验收页
当前虽然有 `doc-boundary-rules.md`，但还缺一类专门资产：
- 报告 vs 请示的对打回归
- 决定 vs 通知的对打回归
- 制度 vs 方案的对打回归
- 通知 vs 通报在不同范围/目的下的切换回归

也就是说：
**有规则了，但“边界题跑一遍”的结果页还不够。**

### 2. rewrite / review / upgrade 三条链还没有吃进部门口径
当前这三份 prompt 已经具备：
- 先判文种
- 再改写/升格/审查
- 不编造事实

但还没有显式要求：
- 判断主责部门；
- 选择对应部门口径；
- 对“行政/人事/后勤/综合运营”的风格差异做收束。

这会导致它们在真实使用中虽然不会串错大文种，但仍可能改出“文种对了、部门味道不够”的中性稿。

### 3. `validation/README.md` 当前标题仍偏“输出模式实稿验收”
现在实际上 validation 已经包含三层：
- 输出模式对打
- 同题不同部门对打
- 真实回归测试矩阵/结果

README 的定位还可以继续升级，不然总入口会落后于资产现状。
