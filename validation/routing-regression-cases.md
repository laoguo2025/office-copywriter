# routing-regression-cases（真实回归测试矩阵）

## 目的
用一组高混淆真实题目，验证 `office-copywriter` 是否已经不只是“资料齐”，而是能够在真实调用时：
- 默认选对文种；
- 默认选对输出模式；
- 默认选对主责部门口径；
- 遇到边界题时，至少知道该如何判断，而不是乱写成看起来很正式的错文种。

---

## 使用方式
每个题目至少做四步判断：
1. **文种判断**
2. **输出模式建议**
3. **主责部门判断**
4. **容易翻车点**

如需扩展到真实执行，可继续补：
- 推荐读取链路
- 起草 prompt
- 一版最小可用成稿

---

## Case 01｜清明节期间值班及节后工作衔接安排
- 场景描述：需要发给各部门，明确放假前收尾、假期值班、节后返岗、材料报送与应急联络要求。
- 预期文种：**通知**
- 预期输出模式：**正式发文版**
- 预期主责部门：**行政**
- 容易翻车点：容易被写成综合运营口径的“推进通知”，导致闭环要求压过周知执行。
- 快速判断依据：核心是统一安排、周知执行、报送衔接，不是专项推进闭环。

## Case 02｜一季度重点任务推进滞后，要求各责任部门重排节点并反馈风险
- 场景描述：面向多部门，要求对滞后任务重新拆解节点、补充风险说明，并按周反馈推进情况。
- 预期文种：**通知**
- 预期输出模式：**正式发文版**
- 预期主责部门：**综合运营**
- 容易翻车点：容易套成行政通知，写成“请各部门按时反馈”，但缺少节点、台账、闭环机制。
- 快速判断依据：核心是推进任务、跨部门协调、节奏管理和闭环反馈。

## Case 03｜员工安全培训签到表与实际到场人数不一致，需要向领导说明原因和后续措施
- 场景描述：不是请求审批，而是对异常情况进行解释，并说明核查、补训和整改安排。
- 预期文种：**情况说明 / 报告型说明**
- 预期输出模式：**领导汇报版**
- 预期主责部门：**人事**（若培训由人事牵头）
- 容易翻车点：容易误写成请示，或者写成纯解释、没有整改措施。
- 快速判断依据：目标是报情况和说明处理，不是请批。

## Case 04｜食堂燃气报警装置巡检中发现隐患，需要说明处置过程、整改进展和后续保障
- 场景描述：向管理层上报现场问题与整改安排，强调已采取的临时措施和后续复检计划。
- 预期文种：**报告**
- 预期输出模式：**领导汇报版**
- 预期主责部门：**后勤**
- 容易翻车点：容易写成通报，导致对上汇报场景被误用成对下警示口径。
- 快速判断依据：核心是向上汇报现场问题与整改进展，不是面向全员做警示传播。

## Case 05｜对近期考勤异常多发部门进行集中点名并要求限期整改
- 场景描述：需要在一定范围内说明情况、指出问题、形成警示，并明确整改要求。
- 预期文种：**通报**
- 预期输出模式：**正式发文版**
- 预期主责部门：**人事**
- 容易翻车点：容易被写成通知，结果只剩“请各部门重视”，警示导向不够。
- 快速判断依据：重点是说明典型情况、形成警示、推动整改，不只是安排执行。

## Case 06｜新增办公用品领用规范，后续长期执行并纳入月度检查
- 场景描述：明确适用对象、领用流程、审批要求、保管责任和监督机制，长期适用。
- 预期文种：**制度 / 管理办法**
- 预期输出模式：**制度条款版**
- 预期主责部门：**行政**
- 容易翻车点：容易写成“工作方案”，导致只有推进动作，没有稳定规则。
- 快速判断依据：核心是长期规范，不是阶段专项。

## Case 07｜二季度新员工入职培训专项安排，覆盖目标、对象、课程、分工、时间表和保障
- 场景描述：围绕某一阶段性工作做系统安排，需要明确组织方式和推进节奏。
- 预期文种：**方案**
- 预期输出模式：**正式发文版**
- 预期主责部门：**人事**
- 容易翻车点：容易和制度混淆，写成长期规则；也容易和工作计划混淆，写得过于概括。
- 快速判断依据：核心是专项、阶段性推进，不是长期约束，也不只是日常计划罗列。

## Case 08｜申请增加区域项目驻场运维人员编制 2 名
- 场景描述：说明现状压力、岗位缺口、业务影响和拟申请事项，请领导审批。
- 预期文种：**请示**
- 预期输出模式：**领导汇报版**
- 预期主责部门：**人事** 或 **综合运营**（视编制归口）
- 容易翻车点：容易写成报告，导致“写了很多情况但没有明确请批事项”。
- 快速判断依据：核心目的是请批，不是单纯汇报。

## Case 09｜关于调整区域项目例会机制的决定
- 场景描述：明确自何时起调整会议频次、参加范围、议题要求和会议纪律，作为正式组织决定发布。
- 预期文种：**决定**
- 预期输出模式：**正式发文版**
- 预期主责部门：**综合运营**
- 容易翻车点：容易写成通知，导致组织裁定性、正式性不够。
- 快速判断依据：核心是对重要事项作出正式确定，而不是单纯周知。

## Case 10｜周例会原始记录整理成可执行结论，明确责任人和完成时限
- 场景描述：已有现场发言记录，需要整理成可流转、可跟进的正式成果。
- 预期文种：**会议纪要**
- 预期输出模式：**会议落地版**
- 预期主责部门：**综合运营**（若为协调推进会）
- 容易翻车点：容易保留大量过程发言，变成会议记录而非纪要。
- 快速判断依据：核心是提炼议定事项并形成执行依据。

## Case 11｜某部门上报材料连续两周逾期，要求说明原因并立即纠正
- 场景描述：需要指出问题、说明影响、明确整改要求，但不一定上升到全员警示。
- 预期文种：**通知** 或 **通报**（边界案例）
- 默认建议：
  - 若范围小、重点在整改安排 → **通知**
  - 若要形成警示导向、典型曝光 → **通报**
- 预期输出模式：**正式发文版**
- 预期主责部门：**行政** 或 **综合运营**（看归口）
- 容易翻车点：在边界题上不作判断，直接写成泛泛“正式文件”。
- 快速判断依据：是否以“警示导向”为主，是通知/通报的关键分界。

## Case 12｜把一段口语化工作安排整理成能直接发群内的正式内部通知
- 场景描述：用户只给了口语碎片，没有明确文种，但目标是形成可直接下发的规范文本。
- 预期路径：**rewrite-rough-notes → 通知 → 正式发文版或快速可用稿**
- 预期主责部门：按内容判断，若是日常安排通常先偏**行政**
- 容易翻车点：直接润色原话，不先判断文种和部门口径。
- 快速判断依据：这是典型 rewrite 链入口，不是单纯“改正式一点”。

---

## 当前覆盖面
这 12 个 case 主要覆盖：
- 行政通知 vs 综合运营通知
- 人事情况说明 vs 请示 / 报告
- 后勤报告 vs 通报
- 报告 vs 请示
- 决定 vs 通知
- 制度 vs 方案
- 会议纪要 vs 会议记录
- rewrite 链入口判断

---

## 通过标准（建议）
若一个真实调用链想判为“基本通过”，至少应满足：
1. 文种判断正确；
2. 输出模式建议不跑偏；
3. 主责部门判断有依据；
4. 对边界题能说清为什么，不是碰运气；
5. rewrite / review / formalize 链不再绕开文种与部门判断。
