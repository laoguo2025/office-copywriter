# rewrite-modes（非默认）

> 这份文件只用于 **改写 / 转正式 / 审稿** 场景的快速判断，
> 不是默认起草入口。

## 什么时候再看这份文件
只在下面几类任务中进入：
- 用户给的是草稿、口语描述、聊天记录、会议笔记；
- 用户要“润色、改正式、整理成文”；
- 用户不是要从零起草，而是要把已有材料改造成内部文书。

默认起草时不要先读这份文件，优先走：
1. `minimal-invocation.md`
2. `pragmatic-scope.md`
3. `doc-type-routing.md`
4. 对应模板 / drafting prompt

---

## 改写任务的最短判断
### 1) 口语需求 / 零散要求
优先去向：
- 通知
- 请示
- 情况说明

### 2) 会议记录 / 发言要点 / 聊天整理
优先去向：
- 会议纪要
- 会议执行版纪要

### 3) 已有草稿但结构乱
优先去向：
- 请示 / 报告 / 通知 等对应正式文种

### 4) 事实经过 / 异常事件 / 人员问题
优先去向：
- 情况说明
- 通报
- 报告

### 5) 管理要求已经明确，但还没成文
优先去向：
- 制度 / 办法
- 方案

### 6) 已有完整文稿，只是要更正式
优先去向：
- 审稿优化
- 正式化改写

---

## 直接执行时优先用什么
真正动手时，优先进入 prompt，而不是停留在这里：
- `prompts/rewrite-rough-notes.md`
- `prompts/upgrade-to-formal-version.md`
- `prompts/review-doc.md`

如果文种边界不稳，再补读：
- `doc-boundary-rules.md`
- `direction-rules.md`

---

## 使用原则
- 先判断“这是改写任务，不是从零起草任务”；
- 先保留事实和原意，不乱补新结论；
- 先把材料收束成一个可交付版本；
- 缺信息时，输出 `可用改写版 + 待补信息`；
- 不要让这份文件变成又一层长说明书。
