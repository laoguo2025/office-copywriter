# pragmatic-scope（够用版原则）

## 这份 skill 现在应该追求什么
不是追求“覆盖所有企业、所有部门、所有文风差异”。
而是追求：

- **不写错文种**
- **不写错方向**（上行 / 下行 / 平行）
- **能切基础输出模式**
- **能做基础部门口径区分**
- **默认给出一个规范、可用、可流转的内部文稿底稿**

一句话：
**先保证正确和可用，不追求把所有企业个性都预埋进 skill。**

---

## 保留什么
这些能力继续保留，且是这份 skill 的主骨架：

### 1. 文种判断
保留：
- 通知
- 请示
- 批复
- 报告
- 会议纪要
- 通报
- 决定
- 制度 / 管理办法
- 方案
- 工作计划
- 总结 / 情况说明

原因：这是“写不错类型”的基础。

### 2. 输出模式
保留：
- 快速可用稿
- 正式发文版
- 领导汇报版
- 制度条款版
- 会议落地版

原因：这是“同一文种按用途切形态”的高价值能力。

### 3. 基础部门口径
保留：
- 行政
- 人事
- 后勤
- 综合运营

但只保留到“基础风格收束”层，不继续无限细分。

### 4. 少量高价值边界规则
保留：
- 请示 vs 报告
- 决定 vs 通知
- 制度 / 管理办法 vs 方案
- 通知 vs 通报

原因：这些是最容易写错、但最值钱的分界。

---

## 不继续做什么
以下方向，不应再作为默认优化主线：

### 1. 不为不同企业预埋大量细颗粒文风
原因：
- 企业差异很大；
- 很难抽成稳定通用规则；
- 预埋越多，越容易让 skill 变重、变脆、变难维护。

### 2. 不无限扩展部门变体
例如：
- 行政再拆办公室 / 综合部 / 总经办
- 人事再拆招聘 / 培训 / 绩效
- 综合运营再拆经营分析 / PMO / 督办

这些可以在真实使用时临时约束，不必都固化进 skill。

### 3. 不为了“像真人”继续堆大量 reference / validation
“像真人”不是靠文件数量堆出来的。
如果继续堆而不提升默认可用性，就是费力不讨好。

---

## 更实用的默认策略
以后默认这样用：

1. **先用 skill 解决共性问题**
   - 文种
   - 结构
   - 方向
   - 模式
   - 基础口径

2. **再把企业个性留到运行时输入**
例如让用户补一句：
- 更像老板平时口气
- 更短一点
- 更硬一点
- 更像我们公司内部老通知风格
- 少一点官话

也就是说：
**企业差异应主要作为“即时约束”，不是“预埋规则库”。**

---

## 当前阶段结论
`office-copywriter` 到现在已经够用了。
后续更应该做的是：
- 控制复杂度；
- 保持主骨架清晰；
- 让它更容易调用，而不是更像百科全书。

如果必须在“更复杂”和“更好用”之间二选一，
这份 skill 现在应明确站在：

**更好用。**
